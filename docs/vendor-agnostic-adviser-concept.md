# **РЕАЛИЗАЦИЯ VENDOR-AGNOISTIC LLM advisor strategy system**

Идея: 'Anthropic just released the advisor strategy, and it's a big deal for anyone building with Claude. Instead of running every task through Opus, you can pair it as an advisor with a cheaper model like or doing the actual work.'

## **ЦЕЛИ ПРОЕКТА**

Задача — реализовать vendor-agnostic систему для работы с LLM моделями через единый интерфейс, используя абстрактные классы, фабрику агентов, конфигурацию в YAML и CLI-интерфейс.
Фактически:
- planner–executor architecture
- иногда critic loop (проверка результата)

### Роли в системе

Минимально разумная схема:
**Planner** (advisor) — декомпозиция, стратегия, Цель отделить планирование от исполнения
Роутинг - пока хардкод, в зависимости от задачи.
**GenericExecutor** — универсальный исполнитель. Выполняет общие шаги плана, где результат не обязательно код.
**CodeExecutor** — специализированный для программирования. Выполняет задачи, где результат — исполнимый код
**Critic** (опционально, но желательно) — проверка и коррекция

### Retry стратегия
Если critic недоволен:

НЕ перепланируй сразу
сначала попробуй retry step с другим executor

Упрощенный вариант workflow:
```
User input
   ↓
[Planner LLM]
   → генерирует JSON план:
     - шаги
     - вход/выход каждого шага
     - критерии успеха
   ↓
For each step:
   [Executor LLM]
     → выполняет шаг
     → возвращает результат
   ↓
[Reviewer LLM] (опционально)
   → проверяет
   → либо approve, либо revise
```

### **Основные цели:**

1. Создать архитектуру, которая позволяет:  
   * Работать с разными моделями (Llama, Qwen, Minimax, Mistral, DeepSeek, Gemma) через единый интерфейс.
   * Абстрагировать различия между моделями (например, Minimax использует functions вместо tools).  
   * Гибко настраивать модели через YAML-конфигурацию.  
   * Предоставлять CLI-интерфейс для удобного использования.  
   * Обеспечивать надежность (retry, обработка ошибок, логирование).  
2. Реализовать следующие компоненты:  
   * абстрактный базовый интерфейс агента.  
   * агент для каждой из агента-роли в проекте (Planner, Кодер, Исполненитель, Проверка) реализующий базовый интерфейс.  
   - Загрузка каждым агентом файла prompt из файловой системы.
   * factory.py — фабрика для создания агентов 
   - При создании агентов учитывается среда окружения (dev/test). Например для dev создаются 'Mock' агенты.
   - Параметры создаваемых агентов загружаются из файлов yaml конфигурации. (например название модели, роль, температура и т.п.)
   * config/models.yaml и config/agents.yaml — конфигурация моделей и агентов.  
   * cli.py — CLI-интерфейс с командами ask, chat, list-models.  
   * tests/ — тесты для всех компонентов.  
   * Документацию (README.md).  
3. Обеспечить следующие фичи:
   * Поддержка tools (функций) для всех моделей.  
   * Retry и обработка ошибок (с использованием tenacity).  
   * Логирование (с использованием logging).  
   * Кеширование (с использованием lru\_cache).  
   - Persistance с помощью SQLLight
   * Настройки (хранение API-ключей в .env). 
   - При старте приложения может задаваться среда окружения, например 'dev', в этом случае используется dev.env, в случае 'test', используется test.env
   * Документация (примеры использования, пример .env).

---

Коммуникация с моделями будет через LLM-клиент через LiteLLM proxy на Message API (без streaming)
 
## **ТРЕБОВАНИЯ К ПЛАНУ РЕАЛИЗАЦИИ**

Реализация будет проходить последовательно, небольшими шагами, каждый шаг будет добавлять участок функциональности.

Давай спланируем

Этап 0 — Базовая инфраструктура
- Шаг 0.0. Скелет проекта, запуск, README, Hello world
- Шаг 0.1. Создать базовые конфигуации, старт приложения с параметром переменной среды dev/prod
- Шаг 0.2. Настройка логирования, подключения к базе данных.
- Шаг 0.3. Загрузка конфигов в зависимости от среды выполнения.

Этап 1 — фабрика для LLM агентов (пока mock)
- Шаг 1.1. базовый REPL с выводом информации о приложении.
- Шаг 1.2. Интерфейс/абстрактный класс агента
- Шаг 1.3. Фабрика агентов, создание агента фабрикой согласно конфигурации агента. Каждый агент загружает prompt из файла.
- Шаг 1.4. Реестр агентов
Провести минимальную sequential проверку всего процесса исполнения с mock агентами

Этап 2 — Минимально работающий агент Planning
Цель: Создать минимально работающий реальный агент
Пока НЕ делать: не исполнять шаги автоматически, просто выводить план.

Проверка: планы:
- короткие (≤ 5–8 шагов)
- атомарные
- корректно размечены (type)

Этап 3 — Sequential execution (без routing)
Цель: Научиться выполнять план

- Создать агента исполнителя
- взять план
- выполнять шаги последовательно

ВСЕ шаги → одной моделью

Проверка: pipeline работает end-to-end
нет зависаний
шаги выполняются по порядку

Этап 4 — Введение routing
Цель: Разделить code и text execution

Сделать rule-based routing:
```
if step.type == "code":
    use CodeExecutor
else:
    use GenericExecutor
```

Проверка
каждая модель выполняет свои задачи


Этап 5 — Добавление Critic
Цель

Контроль качества шагов

Сделать
после каждого шага:
отправлять результат в Planner (critic mode)
ответы:
APPROVE
REJECT + причина
Логика
при REJECT → retry шага (1–2 раза)
Проверка
уменьшается количество ошибок
нет бесконечных циклов
